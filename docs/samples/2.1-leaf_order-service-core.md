# 叶子模块分析：`order-service` / `service-core`

> 编号 **2.x**：由第一层产物派生的叶子/深化产出。样例文件。对应 `1.3-module_tree.json` 中的 `leaf-service-core`。

## 1. 范围与边界

| 项 | 值 |
|----|-----|
| 路径 | `order-service/src/main/java/com/example/order/service` |
| Maven 模块 | `order-service` |
| 粗粒度 LOC（Java） | 12 000（演示数据） |
| 解析置信度 | 高（手写源码为主）；含少量 Lombok，生成成员未逐条展开 |

## 2. 职责摘要

本叶子覆盖「订单应用服务」层：编排领域对象、事务边界、与外部网关/仓储交互。对外主要由 `OrderController`（`order-web`）经接口注入调用；对内依赖 `order-domain` 聚合根与值对象。

## 3. 关键类型（节选）

| FQN | 角色 | 备注 |
|-----|------|------|
| `com.example.order.service.OrderService` | 应用服务 | 事务入口；调用 `Order.validate()` |
| `com.example.order.service.PricingFacade` | 防腐层 | 委托 SPI `PricingProvider` |
| `com.example.order.service.audit.AuditHelper` | 横切 | 2018 年后逐步被 AOP 替代 |

## 4. 对外依赖（出边摘要）

- **模块**：`order-domain`（领域模型）、`order-integration`（虚构，未在迷你图中展开）
- **框架**：Spring `@Transactional`、Hibernate Session（遗留 `sessionFactory` Bean 名）

## 5. 风险与历史债（静态可观测）

1. **事务边界过大**：`OrderService#placeOrder` 方法圈复杂度高，含多分支与补偿逻辑（建议结合调用图细化）。
2. **同步远程调用**：`PricingFacade` 内仍存在阻塞 HTTP 调用路径；与当前异步化目标不一致。
3. **已弃用 API**：`LegacyOrderFormatter` 仍被同包测试引用，生产路径标记为 `@Deprecated` 但未删除。

## 6. 建议阅读顺序（ onboarding）

1. `OrderService` 公共方法签名与 DTO 映射  
2. `order-domain` 包内 `Order` 状态机  
3. `META-INF/services` 定价扩展点与实现类对照表（见仓库根 `docs` 遗留说明）
